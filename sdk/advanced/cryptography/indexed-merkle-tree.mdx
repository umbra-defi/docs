title: "Indexed Merkle Tree"
description: "The on-chain Stealth Pool Note commitment store, with effectively infinite capacity."
---

The Stealth Pool uses an **Indexed Merkle Tree** (IMT) — a variant of the standard Merkle tree that supports efficient membership proofs and nullifier range checks. See the [Unified Mixer Pool](https://docs.umbraprivacy.com/docs/core-concepts/unified-mixer-pool) and [Unlocking Address Strategies](https://docs.umbraprivacy.com/docs/transaction-lifecycle/unlocking-address-strategies) pages for higher-level context.

## Structure

- Maximum depth: 20 (up to 2^20 ≈ 1,048,576 leaves per tree).
- Up to 2^128 trees can exist per pool — new trees are created as existing ones fill up.
- Leaf insertions are append-only and monotonically indexed.
- Each leaf stores a Stealth Pool Note commitment.

The total theoretical capacity across all trees is 2^20 × 2^128 = **2^148 notes**. At a sustained rate of one billion notes per second, it would take approximately 10^28 years to exhaust a single pool's capacity. In practice, the pool is effectively infinite.

## Insertion

When a Stealth Pool Note is written, its Poseidon commitment is appended to the current active tree. The insertion index is embedded in the ZK proof input, binding the proof to that specific leaf position.

## Burning and per-batch Merkle proofs

When a note is burnt, the burner factory fetches a per-batch Merkle inclusion proof (20 sibling hashes + current root) from the indexer **immediately before submitting the burn**. The proof is embedded in the Groth16 circuit input and verified on-chain.

The on-chain program stores a rolling buffer of the **100 most recent Merkle roots**. A ZK proof generated against any of these roots is valid on submission — you do not need to prove against the absolute latest root. This gives a practical window between proof generation and transaction submission, accommodating network latency, wallet confirmation delays, and slower clients.

<Note>
The scanner does not pre-bundle Merkle proofs. Proofs are fetched per batch at burn time so a single proof set is shared across an entire batch — far cheaper than fetching one proof per note. Callers should pass scanned notes directly to a burner factory; `enrichWithMerkleProof` is exported from `@umbra-privacy/sdk/burn` only for callers building a fully custom burn pipeline.
</Note>

<Warning>
Merkle proofs become stale once the root they were generated against is evicted from the 100-root buffer. In high-throughput conditions where many notes are inserted rapidly, this window can shorten. The burner factory refetches the proof on each retry, so simply re-running a failed burn is the recommended recovery path.
</Warning>
