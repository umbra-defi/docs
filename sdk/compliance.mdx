## Overview

Umbra's compliance system is entirely voluntary and user-initiated. Two independent mechanisms cover different parts of the protocol:

<CardGroup cols={2}>
  <Card title="Stealth Pool Viewing Keys" icon="key" href="/sdk/compliance-viewing-keys">
    Derive scoped cryptographic keys from your master viewing key and share them with auditors. Viewing keys give read access to Stealth Pool Note activity within a specific time window or token scope ã without exposing anything outside that scope. Pure off-chain Poseidon hierarchy.
  </Card>
  <Card title="X25519 Compliance Grants" icon="lock-open" href="/sdk/compliance-x25519-grants">
    Create on-chain grants that authorise Arcium MPC to re-encrypt your EncryptedTokenAccount balances under a grantee's X25519 key. The grantee then decrypts the re-encrypted ciphertexts with their own private key.
  </Card>
</CardGroup>

<Warning>
The two mechanisms cover **different data**. Stealth Pool viewing keys only decrypt the Poseidon `pc_encrypted_*` fields on a note (the mixer pool ciphertext). X25519 grants only cover the Rescue `rc_encrypted_*` ciphertext on an ETA. **Viewing keys cannot decrypt ETA balances; X25519 grants cannot decrypt notes.** Pick the right tool for the data you are auditing.
</Warning>

<Note>
No party can access your data without your explicit action. Neither mechanism is reversible for data the grantee has already received ã revoking a grant prevents future access but does not invalidate ciphertexts already re-encrypted.
</Note>

Each level is derived from the level above using a Poseidon hash. Sharing a key at any level grants read access to all note activity within that scope and nothing outside it. The Second Viewing Key (a.k.a. the transaction viewing key ã TVK) is the one that actually decrypts a note's Poseidon stream cipher (`pc_encrypted_*`).

Eight derivers ship, all from the SDK's key-derivation primitives:

## X25519 compliance grants

X25519 compliance grants use on-chain PDAs to authorise Arcium MPC to re-encrypt your ETA ciphertexts under a grantee's X25519 key. You create the grant voluntarily, choose the receiver and a unique nonce, and can revoke it at any time.

```typescript
import { getComplianceGrantIssuerFunction } from "@umbra-privacy/sdk/compliance";

const createGrant = getComplianceGrantIssuerFunction({ client });

// Create a grant allowing `receiver` to re-encrypt your shared ETA ciphertexts.
// Arguments are positional: (receiver, granterX25519, receiverX25519, nonce, optionalData?, hooks?)
const result = await createGrant(
  receiver,        // receiver's wallet address (Address)
  granterX25519,   // your MVK X25519 public key (X25519PublicKey)
  receiverX25519,  // receiver's X25519 public key (X25519PublicKey)
  nonce,           // u128 bigint nonce (RescueCipherEncryptionNonce)
);
```

See [X25519 Compliance Grants](/sdk/compliance-x25519-grants) for the complete workflow ã deriving the granter's MVK X25519 keypair, generating a random nonce, re-encrypting ciphertexts under the grant, and querying grant status.

- `getComplianceGrantRevokerFunction` ã revoke (deletes the grant PDA).
- `getUserComplianceGrantQuerierFunction` ã query a user-issued grant.

- `getNetworkCiphertextReencryptorForNetworkGrantFunction` ã re-encrypt under a network grant.
- `getSharedCiphertextReencryptorForNetworkGrantFunction` ã re-encrypt a Shared-mode ciphertext for a network grantee.


## Privacy and Trust Model

- Compliance grants are stored on-chain as marker PDAs ã their existence is transparent and auditable.
- A viewing key at scope X gives access only to data within scope X ã no access to anything outside that scope.
- Revoking a compliance grant (deleting the PDA) prevents future re-encryption requests, but does not invalidate ciphertexts the grantee has already obtained.
- The master viewing key and master seed are never shared directly through either mechanism.

<Warning>
Rescue is a **stream cipher**. Never reuse a grant nonce ã doing so leaks the XOR of two plaintexts. Use `generateRandomNonce` from `@umbra-privacy/sdk/arcium` for fresh nonces, and persist them per `(granter, receiver, mint)` if you need to re-issue with the same parameters.
</Warning>
